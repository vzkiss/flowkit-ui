This page is a **field note** from building a **Template** feature: users pick an existing template by name, create a new one from the typed query, then save, update, or delete the current template. The combobox is only the **name** control; the buttons are your app’s persistence and validation.

## The problem

shadcn/ui ships a solid Combobox, but not a “creatable” variant out of the box. The usual alternative is `react-select` with `Creatable`—which works, yet often diverges from your design system (spacing, borders, dark mode, focus). Wiring a third-party creatable into shadcn-like markup is ongoing maintenance.

`CreatableCombobox` in Flowkit keeps the same stack as the rest of the UI: **Combobox + Base UI**, with a small wrapper for the create row and `onCreateValue`.

## What it looks like in the product

Search and create in one list: type a query, choose **Create "…"** if nothing matches, or pick an existing row.

![Template name field with Create option and existing templates]

When an existing template is selected, expose **Update** and **Delete** (and any overflow actions). When the user is creating or editing a **new** name before it exists on the server, **Save** (or “Create template”) is the right primary action instead of update.

## Pattern sketch (state, not a full app)

Keep **three concerns** separate:

1. **Options** — list of templates from the server (`items`).
2. **Selection** — what the combobox shows (`value` / selected item id).
3. **Actions** — save new, update existing, delete; enable buttons from (1) and (2).

Rough shape:

```tsx
// items: { id: string; value: string; label: string }[]
// selectedId: string | null — set when user picks an existing row
// draftName: string — controlled input / combobox text for "new" or rename

function canSave() {
  // e.g. draftName trimmed, no duplicate, not already persisted
}

function canUpdate() {
  return selectedId != null && draftName matches that row
}

function canDelete() {
  return selectedId != null
}
```

- **Save** — call when the user confirmed a **new** name (`onCreateValue` and/or explicit save): append to `items`, persist, then set `selectedId` to the new id.
- **Update** — when `selectedId` is set and the label changed: patch server, update `items`.
- **Delete** — when `selectedId` is set: remove from server, clear selection, trim `items`.

The combobox does **not** decide which button to show; it only reports selection and creation via `onValueChange` and `onCreateValue`. Your screen maps those to API calls and button disabled states.

### reddit note

Reach for this when users should pick from a **named list** (templates, tags, categories, saved views) **or** type a new name in the same control—without a separate “add new” screen or modal.

It composes the shadcn/ui Combobox and Base UI so markup and styling stay aligned with the rest of your app. That avoids pulling in `react-select` Creatable (or similar) and fighting to match tokens, focus rings, and density.
